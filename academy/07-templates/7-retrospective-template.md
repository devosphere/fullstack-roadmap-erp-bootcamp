# Purpose

A Sprint Retrospective is a structured team activity performed at the end of every Sprint to inspect how the team worked and identify opportunities for improvement.

The goal is not to assign blame. The goal is continuous improvement.

The team should answer:

- What went well?
- What did not go well?
- What should we improve?
- What actions will we take in the next Sprint?

# What Did Not Go Well?

Identify problems or challenges.

Examples:

- Requirements were unclear.
- Tasks were underestimated.
- Testing started too late.
- Technical decisions were delayed.
- Dependencies blocked progress.

Focus on process improvement, not individual blame.

# Root Cause Analysis

For significant problems, investigate the root cause.

## Problem

Describe the issue.

---

## Why Did It Happen?

Use the Five Whys approach.

Example:

### Why?

Deployment failed.

### Why?

Environment variables were missing.

### Why?

Configuration was not documented.

### Root Cause:

Environment configuration management process was incomplete.

# GitHub Traceability

Retrospectives should connect improvement actions back into project execution.

```text
Sprint Retrospective
        │
        ▼
Improvement Action
        │
        ▼
GitHub Issue
        │
        ▼
Sprint Planning
        │
        ▼
Implementation
        │
        ▼
Validation
        │
        ▼
Release
```

Improvement items should become actual tracked work, not just discussion points.

---

# Definition of Done

A Sprint Retrospective is complete when:

- [ ] Sprint results are reviewed.
- [ ] Successes are identified.
- [ ] Problems are documented.
- [ ] Root causes are analyzed.
- [ ] Improvement actions are created.
- [ ] Owners are assigned.
- [ ] Actions are tracked in GitHub.
- [ ] Improvements are reviewed in the next Sprint.
